# Project Plan: Bootstrap FluxCD for GitOps tracking

This document details the plan to bootstrap and configure **FluxCD** to manage the home lab Kubernetes cluster using GitOps principles. We will configure Flux to sync with your self-hosted **Gitea** instance.

---

## 1. Prerequisites

1. **Gitea Repository**: Create a private repository in your Gitea instance (e.g., `nick/fog-k8s` or migration of the current `fog` repository).
2. **Access Token/SSH Key**:
   * We will generate a dedicated SSH deploy key so Flux can pull/push configuration updates securely to Gitea.
3. **Flux CLI**: Install the Flux CLI on your local machine.

---

## 2. Directory Layout Architecture

We will structure the GitOps repository using the standard Flux monorepo structure:

```text
.
├── clusters/
│   └── home-lab/
│       ├── flux-system/            # Auto-generated by Flux Bootstrap
│       │   ├── gotk-components.yaml
│       │   └── gotk-sync.yaml
│       ├── infrastructure.yaml     # Points to /infrastructure
│       └── apps.yaml               # Points to /apps
│
├── infrastructure/
│   ├── base/                       # Shared platform components
│   │   ├── cert-manager/
│   │   ├── ingress-nginx/
│   │   └── external-dns/
│   └── sources/                    # HelmRepositories and GitRepositories
│
└── apps/
    ├── base/                       # Application Deployments
    │   └── gitea/
    │       ├── release.yaml        # HelmRelease
    │       ├── values.yaml         # Values overrides
    │       └── secrets.yaml        # ExternalSecret definition
    └── production/                 # Environment overrides
```

## 3. Bootstrapping Flux on Gitea

We will use the generic Git provider bootstrap because Gitea supports standard Git SSH protocols.

### Step 3.1: Run the Bootstrap Command
Export your Gitea credentials and run the bootstrap command. Flux will automatically generate an SSH deploy key, configure it inside Gitea (if using HTTP token), or prompt you to add the public deploy key to Gitea manually.

```bash
# Set up environment variables
export GITEA_TOKEN="<your-gitea-personal-access-token>"
export KUBECONFIG="./kubeconfig"

# Run Flux bootstrap
flux bootstrap git \
  --url=ssh://git@gitea.fog.chalko.com/nick/fog-k8s.git \
  --branch=main \
  --path=clusters/home-lab \
  --password=$GITEA_TOKEN
```
*(Alternatively, if bootstrapping without a token, Flux will generate a local SSH key and print the public key. You will manually add it as a Deploy Key in the Gitea repository settings).*

## 4. Migrating Existing Apps to HelmRelease

Instead of running `helm install` commands manually, we will define them as `HelmRelease` manifests. 

For example, our Gitea installation will look like this:

### `apps/base/gitea/release.yaml`
```yaml
apiVersion: helm.toolkit.fluxcd.io/v2
kind: HelmRelease
metadata:
  name: gitea
  namespace: gitea
spec:
  interval: 10m
  chart:
    spec:
      chart: gitea
      version: "12.6.0"
      sourceRef:
        kind: HelmRepository
        name: gitea-charts
        namespace: flux-system
  valuesFiles:
    - values.yaml
```

## 5. Execution Roadmap

1. **Create the repository** `nick/fog-k8s` in your Gitea instance and push the current repository content.
2. **Install Flux CLI** on your local machine:
   ```bash
   curl -s https://fluxcd.io/install.sh | sudo bash
   ```
3. **Run the Flux bootstrap command** pointing to Gitea.
4. **Create Kustomization rules** inside `clusters/home-lab/` to scan `/infrastructure` and `/apps`.
5. **Convert existing releases** (`ingress-nginx`, `external-dns`, `cert-manager`, and `gitea`) into Flux `HelmRelease` manifests.
6. **Commit and push** to Gitea. Flux will take over management and reconcile state.
